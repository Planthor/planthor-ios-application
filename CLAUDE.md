## Commands

```bash
# Install dependencies
flutter pub get

# Run code generation (required after any @riverpod annotation or model change)
dart run build_runner build --delete-conflicting-outputs

# Analyze (must pass zero errors before PR)
flutter analyze

# Run app (copy dart_defines/*.example.json → dart_defines/<env>.json first)
flutter run --dart-define-from-file=dart_defines/dev.json        # dev (Docker Compose)
flutter run --dart-define-from-file=dart_defines/prod.json       # prod (cloud)

# Run all tests
flutter test

# Run single test file
flutter test test/widget_test.dart
```

## Architecture

Feature-first Clean Architecture with Riverpod state management.

**Entry point:** `lib/main.dart` wraps `MyApp` in `ProviderScope`, watches `appThemeProvider`, routes to `SignInScreen`.

**Navigation flow:** `SignInScreen` → `MainScaffold` (bottom nav). `MainScaffold` watches `navigationProvider` (generated Riverpod notifier) to switch between `DiscoveryScreen` and `GardenScreen`.

**Feature structure** — each feature under `lib/features/<name>/`:
```
presentation/
  screens/      # Full-page widgets (ConsumerWidget)
  widgets/      # Feature-scoped UI components
bloc/           # Riverpod providers / state notifiers
domain/         # Entities and use-case logic (Phase 2+)
```

**Core layer** (`lib/core/`):
- `theme/` — `AppColors`, `AppTheme` (Riverpod-generated, Material 3, forest-green palette)
- `services/`, `utils/`, `widgets/` — shared cross-feature code (mostly empty, Phase 2+)

**Code generation:** All `*.g.dart` files are generated by `build_runner`. Never edit them manually. Any file with `@riverpod` annotation or `part '*.g.dart'` directive requires a build_runner run after changes.

## Key Conventions

- All state widgets extend `ConsumerWidget` or `ConsumerStatefulWidget` (never plain `StatefulWidget` for stateful logic)
- Providers use `@riverpod` annotation + code generation — no manual `Provider(...)` declarations
- New feature = use the `/flutter-feature-gen` skill or follow the `lib/features/<name>/` scaffold above
- PR titles use conventional commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`
- Branch naming: `feature/<name>` or `fix/<name>`
- Every new screen requires a widget test in `test/`
